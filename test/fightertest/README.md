# Fightertest sync test

A seeded Beyond All Reason "fightertest" used as the engine's first CI sync test. The `.github/workflows/fightertest.yml` workflow runs four scenarios (bots, tanks, aircraft, pathfinding) in parallel on separate GitHub Actions workers, each on top of a pinned BAR game + map version, and asserts that a deterministic sync-hash file lands in `$SPRING_DATADIR` at the end of each run. Background github issues: beyond-all-reason/RecoilEngine#2910, parent initiative beyond-all-reason/RecoilEngine#2906.

## Startscripts

Each scenario has its own startscript in this directory:

- `bots-startscript.txt` — bot combat (corak vs armpw, Starwatcher 1.0)
- `tanks-startscript.txt` — tank combat (armbull vs armbull, Starwatcher 1.0)
- `aircraft-startscript.txt` — aircraft combat (corvamp vs armhawk, Starwatcher 1.0)
- `pathfinding-startscript.txt` — pathfinding stress test (armcv/armck, 11k units, Jade Empress 1.3)

## Running locally

Build the engine, make sure `spring-headless` and `pr-downloader` are on your PATH or referenced by absolute path, then:

```
pr-downloader --filesystem-writepath ./bar-data \
    --download-game "Beyond All Reason test-29660-e79286c" \
    --download-map  "Starwatcher 1.0"

spring-headless --isolation --write-dir ./bar-data test/fightertest/bots-startscript.txt
```

Substitute the map and startscript for whichever scenario you want to run. The pathfinding test uses Jade Empress 1.3; the other three use Starwatcher 1.0.

A successful run exits 0 at game frame ~2025 (the startscript's trailing `quitforce`) and leaves a sync-hash file in `$SPRING_DATADIR` for CI to pick up.

## Bumping the pinned BAR version

BAR's rapid `test-<N>-<sha>` tags are content-addressed — once published they never change — which is why the CI doesn't need an external version manifest. To move to a new pin you must edit **two** places in the same commit, or the cache will hand you stale data:

1. `gametype=` and `mapname=` lines in the affected startscript(s)
2. The `key:` on the `Restore bar-data cache` step in `.github/workflows/fightertest.yml` — the cache key includes the map name, so tests using different maps get separate caches

Encoding the game/map identifiers into the cache key is deliberate: a stale key silently reusing the old download is the only real footgun here, and tying the key to the strings it describes makes drift impossible to overlook.

## Replacing a startscript

The startscripts were generated by BAR from running the specific benchmark scenarios in the actual BAR lobby. Then the startscript was pulled from the bar data dir. Should a startscript ever need changing, that's the easiest way to generate it.

A trailing `quitforce` needs to be present in the startscript's debugcommands so that the engine actually quits when finished.
